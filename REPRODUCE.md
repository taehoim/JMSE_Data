# Reproducing the JMSE Marine-Safety Study

End-to-end guide to regenerate every dataset, table, and figure in
*"Uncertainty-Aware Short-Horizon Warning of Large-Inclination Exceedance in
Small Fishing Vessels: A Simulation-Based Multi-Model Benchmark"* (MDPI *Journal
of Marine Science and Engineering*).

> **Notes for this release.**
> - The **manuscript source/PDF is not included** (it is under journal review). This repository
>   reproduces the underlying *dataset, tables, and figures*; any `make paper` / `paper/…`
>   reference below pertains to the authors' full build tree and can be ignored — the released
>   tables and figures are the `*.csv` / `*.png` files under `results/`.
> - **Stage 1 is optional.** The pre-generated dataset is included (`data_processed/curated.parquet`
>   and the raw CSVs under `00_Ref/논문코드_데이터셋/fishingVessel/`), so you can start at Stage 2.
> - Stage 1 requires the third-party **MSS toolbox**
>   ([cybergalactic/MSS](https://github.com/cybergalactic/MSS)), which is **not redistributed
>   here**; install it at `00_Ref/MSS_library/`. The simulation drivers (`regen_6dof.m`,
>   `calcXacc_regen.m`, `fishingVessel_unclamped.m`) are provided in the `fishingVessel/` folder.

The pipeline has three stages:

| Stage | What it produces | Tooling | Time |
|-------|------------------|---------|------|
| **1. Data generation** | the 90-series 6-DOF motion dataset (+ unclamped 10 t variant) → `data_processed/curated.parquet` | MATLAB R2024b (or Octave) + Python | ~1–2 h (MATLAB) |
| **2. Experiments** | per-seed prediction artifacts under `results/**` | Python + CUDA GPU | several hours (5-seed) |
| **3. Analysis & assembly** | every table CSV and figure PNG (T1–T22, F2–F14) | Python (no GPU) | minutes |

The committed result CSVs and figures already correspond to the manuscript, so
you can **skip to Stage 3** to re-derive the post-hoc statistics, or run
`make analysis` directly, without a GPU. Stages 1–2 are only needed to
regenerate the predictions (and the raw data) from scratch.

> **What is and is not in this git repository.** Code, scripts, configs, the
> regeneration `.m` files, the small per-seed result CSVs, and the figure PNGs
> are committed. The bulky regenerable artifacts are **not**: the raw 6-DOF CSVs
> under `00_Ref/`, the processed `data_processed/*.parquet`, per-window
> predictions (`results/**/preds/`, `results/**/*.npz`), and model checkpoints
> (`results/**/*.pt`). These are reproducible from the steps below and are also
> archived on Zenodo (DOI `<ZENODO_DOI>`). See [Obtaining the large artifacts](#obtaining-the-large-artifacts).

## 0. Environment

- **Python 3.12** (the committed results used 3.12.3).
- A **CUDA GPU** for Stage 2 (training). CPU-only works but is much slower.
- **MATLAB R2024b** (`matlab.exe`) for Stage 1 — the engine that produced the
  published data. GNU **Octave** also runs the generation scripts unchanged
  (they are kept engine-neutral); a parity gate against the published CSV is
  provided (`regen_6dof_parity.m`).

```bash
python3 -m venv .venv && source .venv/bin/activate
# exact versions used for the committed results (pins torch==2.6.0 + cu124, numpy==1.26.4, ...):
python3 -m pip install -r requirements-lock.txt
# or loose bounds:
python3 -m pip install -r requirements.txt
python3 -m pytest -q          # sanity check (expect the suite to pass)
```

## Stage 1 — Data generation (MATLAB/MSS)

The motion data are generated by the MSS toolbox driver under
`00_Ref/논문코드_데이터셋/fishingVessel/`. Two regeneration scripts (force-added
to git so they ship even though `00_Ref/` is otherwise ignored) and one EOM
variant drive this stage; the exact recipe and the published simulation config
are documented in [`docs/MATLAB_REGEN_SPEC.md`](docs/MATLAB_REGEN_SPEC.md).

| Script (git-tracked under `00_Ref/…/fishingVessel/`) | Role |
|---|---|
| `regen_6dof.m` | Re-implements the published `generate6Dof_dataset.m` loop, extended to emit **6 wave-phase realizations** (`r0…r5`) per (tonnage, Hs) cell. `r0` carries the original published seed. Config locked to the published values: Modified PM spectrum, `T0=8 s`, heading `140°`, zero speed, `Hs∈{3,5,7}`, `ton∈{10,20,30,40,50}`, `h=1 s`, `T_final=5000` (5001 samples). |
| `regen_6dof_parity.m` | Single-file (10 t, Hs5, r0) parity driver for the regression gate against the published CSV. |
| `calcXacc_regen.m` | Port of `calculateXacc_dataset.m`: adds `Xacc=√(φ²+θ²)` and `GZ=GM·sin(Xacc)`; the `GM_FLOOR` switch selects the clamped (0.30 m) or unclamped (0) hydrostatic floor. |
| `fishingVessel_unclamped.m` (under `00_Ref/MSS_library/VESSELS/models/`) | EOM variant with the transverse-`GM_T` floor removed inside the restoring term — used for the R3.4 GM-floor study via the `USE_UNCLAMPED_EOM` flag in `regen_6dof.m`. |

### 1a. Multi-realization clamped set (the dataset of record)

```bash
cd "00_Ref/논문코드_데이터셋/fishingVessel"
# 90 motion CSVs -> 6Dof_dataset_multi/
matlab.exe -batch "addpath(genpath('../../MSS_library')); regen_6dof" 2>&1 | tr -d '\r'
# add Xacc/GZ with the published 0.30 m GM floor -> 6Dof_dataset_multi_withXacc/
matlab.exe -batch "INPUT_DIR='6Dof_dataset_multi'; OUTPUT_DIR='6Dof_dataset_multi_withXacc'; GM_FLOOR=0.30; calcXacc_regen" 2>&1 | tr -d '\r'
cd -
```

### 1b. Unclamped 10 t variant (R3.4 GM-floor study)

The 10 t hull is the only one whose true `GM = 0.258 m` hits the 0.30 m floor.
Re-simulate its motion with the floor removed *inside the EOM* (the
scientifically valid unclamped path):

```bash
cd "00_Ref/논문코드_데이터셋/fishingVessel"
# re-sim with the unclamped EOM -> 6Dof_dataset_multi_unclampedEOM/
matlab.exe -batch "addpath(genpath('../../MSS_library')); USE_UNCLAMPED_EOM=true; regen_6dof" 2>&1 | tr -d '\r'
# add Xacc/GZ with NO floor -> 6Dof_dataset_multi_unclampedEOM_withXacc/
matlab.exe -batch "INPUT_DIR='6Dof_dataset_multi_unclampedEOM'; OUTPUT_DIR='6Dof_dataset_multi_unclampedEOM_withXacc'; GM_FLOOR=0; calcXacc_regen" 2>&1 | tr -d '\r'
cd -
```

> **Octave** users: replace `matlab.exe -batch "<code>"` with
> `octave --eval "<code>"`. Run `regen_6dof_parity` first and compare its single
> file against the committed published CSV to confirm bit-level parity before
> running the full sweep.

### 1c. Curate into the analysis dataset

The Python pipeline reads the `*_withXacc.csv` files and builds the cached
parquet plus the dataset-summary tables (T2). Curation is **lazy** — it runs
automatically the first time any experiment calls `load_curated()` — but you can
trigger it (and emit the T2 tables) explicitly:

```bash
python3 -c "from jmse.data.curate import build_curated_dataset, dataset_summary; \
            build_curated_dataset(force=True); dataset_summary()"
# writes:
#   data_processed/curated.parquet
#   data_processed/T2_dataset_summary.csv      (realization-averaged per-cell summary -> T2)
#   data_processed/T2_realization_counts.csv   (per-realization row coverage; 3 divergent 10t/Hs7 series truncated)
```

The unclamped directory is overridable via the `JMSE_UNCLAMPED_DIR` env var; by
default `jmse/config.py` resolves it to `6Dof_dataset_multi_unclampedEOM_withXacc/`.

## Stage 2 — Experiments (GPU)

Regenerate every per-seed prediction artifact (the inputs to Stage 3). The full
5-seed run is wrapped in one script:

```bash
bash scripts/run_phase3.sh      # GPU; several hours. Continue-on-error; per-step EXIT logged.
```

This runs, in order: the benchmark sweep, UQ, early-warning alarm, k-of-n
hysteresis, OOD (LOHO/LOVO/LORO) + combined holdout, OOD-UQ, representation,
ablation, and the GM-floor study, followed by the non-GPU analysis steps. It is
re-runnable: each entrypoint overwrites its own `results/` outputs.

Also generate the classical feature baselines (Ridge / gradient boosting),
which back several T4 rows:

```bash
python3 -m jmse.models.feature_baselines
```

Individual GPU entrypoints (see the `experiments` target in the `Makefile`) can
be run standalone, e.g. `python3 -m jmse.sweep --seeds 0 1 2 3 4`.

## Stage 3 — Tables and figures

Once Stage 2 has written the predictions (or using the **committed** result
CSVs, which need no GPU), regenerate the post-hoc statistics and figures:

```bash
make analysis     # UQ rigor, event early-warning, sensitivity, bootstrap CIs, significance
make figures      # regenerate F12/F13/F14 and copy them into paper/figures/
```

## Obtaining the large artifacts

The raw 6-DOF CSVs, the processed parquet, per-window predictions (`*.npz`,
`preds/`), and model checkpoints (`*.pt`) are **not** in git (they are large and
fully regenerable). Two ways to get them:

1. **Regenerate** — run Stage 1 (raw + parquet) and Stage 2 (predictions /
   checkpoints) as above.
2. **Download** the archived snapshot from Zenodo (DOI `<ZENODO_DOI>`) and unpack
   it into the repo root so that `data_processed/` and `results/**/` are
   populated; then run only Stage 3 (`make analysis`).

The committed small result CSVs and figure PNGs are sufficient to re-derive
every post-hoc number and to rebuild the manuscript without the large artifacts.
